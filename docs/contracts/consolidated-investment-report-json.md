# Consolidated investment report JSON contract

`tracky reports investments --db PATH --from YYYY-MM-DD --to YYYY-MM-DD --json` is a read-only inclusive date-range query. It returns schema `tracky.investment-report.v1`; invalid dates and inverted ranges return stable validation errors without opening the database for writes.

The stable top-level sections are `capital_external`, `acquisitions_and_reinvestment`, `returns_and_income`, `costs_and_withholding`, `closing_positions`, and `pending_and_reconciliation`. Every monetary total is `{currency, amount_minor}` and arrays remain grouped by currency; Tracky performs no FX, stablecoin parity, or implicit conversion. Exact quantities remain canonical decimal strings.

External contributions come only from confirmed canonical investment contributions. Brokerage deposits and later purchases therefore cannot count the same capital twice. Gross acquisitions describe asset purchases/constitutions, not new capital. CDT unchanged renewal principal is excluded from external capital but remains a maturity-funded gross renewal; additional principal and capitalized interest remain separately attributable. Returned principal is never income. Interest, dividends, realized results, fees, withholding, other deductions, and net cash remain separate components from active CDT and brokerage operation heads.

Closing state uses the latest snapshot whose provider-effective date (or observation date) is on or before `--to`; later observations are excluded. Reconciliation exposes current and original baseline differences. If no applicable snapshot exists, derived quantity and historical cost still appear with `no_snapshot`, `unavailable`, and a missing-valuation pending item.

Pending/partial contribution allocations, pending provider events, missing snapshot positions, unreconciled differences, missing values, and stale values are never suppressed.

Direct allocation revisions carry an explicit `effective_date` (`YYYY-MM-DD`). Gross acquisitions and closing positions use that economic date rather than the contribution date or `created_at`. Existing databases are migrated with a nullable column: legacy revisions remain undated, are excluded from dated acquisitions/positions, and appear in `pending_and_reconciliation.undated_acquisition_ids` until an append-only correction supplies a date. Replacements may change the date while the allocation is unconsumed; a CDT-consumed allocation keeps its date and economic funding fields immutable.

`acquisitions_and_reinvestment.by_instrument` preserves account, canonical instrument identity, instrument type, cost currency, and minor-unit acquisition amount, so fiat USD and USD-denominated USDC/COPW remain distinct. Direct unconsumed allocations are acquisitions funded by their external contribution. Brokerage buys persist an exact funding split across external capital, existing cash, and reinvested funds; an omitted legacy split is unattributed rather than inferred from cash order. The split follows historical cost through partial sales using brokerage's deterministic moving-average method, and a final sale receives every remaining minor unit. Withdrawals consume income before principal and reduce `net_external_contribution` only for attributed external principal; unresolved portions appear in `pending_and_reconciliation.unattributed_withdrawals`. CDT renewals expose the complete renewed principal as gross acquisition, unchanged matured principal as maturity-funded reinvestment, capitalized interest as reinvestment, and only explicit additional capital as externally funded.

`returns_and_income.by_instrument` and `costs_and_withholding.by_instrument` preserve account and nullable canonical instrument identity. Their per-currency sums reconcile exactly with the monetary totals. Cash activity without an instrument uses `instrument_id: null`; it is never assigned to an arbitrary asset. Instrument identity, not denomination alone, keeps USD fiat, USDC, COPW, and other USD-denominated assets separate. No section performs FX or assumes stablecoin parity.

The query opens SQLite with read-only flags. It neither applies migrations nor writes freshness, baseline, or report state.
