# 0050 — Define the dashboard verification and release contract

Labels: `wayfinder:grilling`

Status: closed

Assignee: yersonargotev

## Parent map

- [Wayfind the local analytics dashboard](0045-wayfind-local-analytics-dashboard.md)

## Question

Which measurable verification and release gates must the adopted dashboard pass
before Tracky may ship it, including functional and semantic correctness,
loopback security, lifecycle cleanup, supported browser baselines, accessibility,
performance and resource budgets, dependency licenses, binary-size impact, and
archive/asset behavior on every Cargo Dist target?

Resolve the exact pass/fail evidence and ownership of each gate so an
implementation agent cannot substitute vague confidence for release proof.

## Blocked by

- [Choose the dashboard product and architecture contract](0049-choose-dashboard-product-and-architecture-contract.md)

## Resolution

Adopt two mandatory evidence layers. Every pull request runs deterministic
product gates; every release candidate validates the real Cargo Dist artifacts
on `aarch64-apple-darwin`, `x86_64-apple-darwin`, and
`x86_64-unknown-linux-gnu`. The implementation author owns tests and fixtures,
CI executes and retains their results, and the publishing maintainer reviews the
per-target evidence and signs the manual checks. One person may fill both roles
but may skip neither. A failed, missing, or waived gate blocks publication;
changing a threshold or accepting a deviation requires an explicit contract
change before release.

### Correctness and read-only behavior

Maintain a deterministic conformance corpus whose expected results are written
manually rather than generated by production calculations. It must cover monthly
metrics, zero-filled periods, filters, canonical pagination, distinct currencies,
income, expenses, savings, investment contributions, historical-cost positions,
valuation freshness, reconciliation alerts, checked numeric limits, and every
valid empty, unavailable, stale, incompatible, and error state. Exact monetary
and decimal JSON strings, aggregate resources, drill-downs, and corresponding
canonical CLI reports must agree exactly.

End-to-end cases cover initial load, filters, successful refresh, failed refresh
that retains the last snapshot, and invalidation of an obsolete drill-down. Each
case compares exact expected data. Schema, rows, and SQLite files must remain
unchanged, and no dashboard path may create a database or run a migration.
Coverage percentages supplement but never replace this scenario matrix. Any
mismatch or write fails the gate.

### Loopback security and lifecycle

Run adversarial integration tests against the spawned release process. It must
listen only on literal `127.0.0.1` with an OS-assigned port and use a distinct,
cryptographically generated capability carrying at least 128 bits of entropy per
process. Missing or wrong capabilities, invalid Host values, cross-site or
invalid Fetch Metadata, disallowed methods, manipulated paths, traversal, and
unknown resources must reveal no financial data. Required CSP, frame, referrer,
MIME, same-origin, and `no-store` policy applies to success and error responses;
there is no CORS opt-in. A secret-bearing decoy fixture must detect leaks.

Loading, navigation, charts, and refresh must make zero external requests. Logs
and errors expose no SQL, database path, private provenance, or capability; the
one intentional exception is the complete startup URL printed for the user.
The publishing maintainer performs a focused review of these controls.

The URL is printed only after the listener accepts connections. Browser-open
failure is non-fatal; startup, database-open, initial-read, and compatibility
failures exit nonzero without leaving a listener. Ctrl-C and SIGTERM complete
graceful shutdown within two seconds. After exit there is no accepting socket,
Tracky descendant, daemon, PID file, temporary file, or extracted runtime asset.
Concurrent instances use independent ports and capabilities, and Tracky does not
close an independently owned browser. Each target must pass at least 100 CI
start/stop cycles with no orphan and no sustained descriptor growth.

### Browser and accessibility baselines

The initial minimum desktop browsers are Safari 26.0, Firefox 153 ESR, and
Chromium/Chrome 150 on the supported operating systems. Test each minimum and
the latest stable release available when cutting a Tracky release. The common
flow covers load, filters, drill-down, refresh, empty/error/stale states, keyboard
navigation, and responsive layout. If JavaScript or graphical enhancement fails,
semantic tables must still expose exact values, period, currency, freshness, and
alerts. A failure in any supported browser blocks release. Minimum-version changes
are documented contract revisions, not implicit evergreen drift.

Every supported view and state must conform to WCAG 2.2 AA. Automated axe checks
allow zero applicable violations. Manual evidence covers keyboard-only operation,
visible and restored focus, VoiceOver with Safari, Orca with Firefox, 200 percent
zoom, reflow at 320 CSS pixels, AA contrast and pointer targets, reduced motion,
and no color-only meaning. Each chart has a title, description, and exact tabular
equivalent. Refresh, stale, and error changes are announced without unexpected
focus movement, and semantic HTML remains useful when enhancement fails. CI owns
automation; the publishing maintainer signs the manual accessibility matrix.

### Performance and resource budgets

Measure the native extracted binary with `--no-open` after five warmups and over
thirty runs per target. Use a deterministic stress fixture spanning 120 months,
100,000 canonical transactions, three currencies, 50 accounts, 200 categories,
and 100 positions with valuations. All latency limits are p95:

- process readiness: at most 500 ms;
- initial snapshot and manual refresh: at most 1.5 seconds each;
- usable page after navigation: at most 2 seconds;
- drill-down response: at most 250 ms; and
- local filter interaction: at most 100 ms.

The Tracky process may use at most 64 MiB idle RSS, 128 MiB peak RSS during
refresh, one percent of one CPU core while idle, eight threads, and 32 open file
descriptors in steady state. After 100 refresh/drill-down cycles it may leak no
descriptor, and memory growth must remain below the greater of five percent or
8 MiB. CI retains raw samples and per-target summaries. Any breach blocks release
until fixed or this contract is explicitly revised.

### Dependencies, size, and distribution

Pin `cargo-deny` and gate licenses, advisories, sources, yanked crates, and
prohibited duplicate dependencies. Known vulnerabilities, unknown licenses,
unreviewed Git or non-crates.io sources, and terms incompatible with MIT static
binary distribution fail the build. GPL, AGPL, SSPL, Commons Clause, and
noncommercial terms are rejected. MPL or any other license with special
obligations requires an explicit review rather than automatic acceptance.
Generate a Cargo.lock-derived dependency/license inventory and required notices
covering Rust, frontend code, and assets. Every archive carries notices that
match the lockfile used to build it.

Before dashboard implementation, preserve an immutable Cargo Dist baseline from
the last dashboard-free commit, using the same toolchain, lockfile, profile, and
targets used for comparison. Per target, the uncompressed HTML, CSS, and
JavaScript total at most 250 KiB. The extracted binary may grow by no more than
both 2.5 MiB and 20 percent; the compressed archive may grow by no more than both
1 MiB and 20 percent; and the resolved graph may add no more than 60 transitive
packages. Record the baseline commit, toolchain, lockfile hash, exact byte counts,
and deltas. Do not move a baseline to hide a regression. Exceeding a limit first
requires reducing features, assets, or dependencies; replacing the adopted stack
requires reopening the architecture decision.

For each target, verify the real Cargo Dist checksum and require the archive to
contain exactly the `tracky` executable, `README.md`, `LICENSE`, and
`THIRD-PARTY-NOTICES`, with correct architecture and executable permission. From
an empty directory and a sandboxed HOME/config, the extracted binary must work
from paths containing spaces and Unicode without a checkout, frontend directory,
network, Node, or auxiliary runtime file. Embedded endpoints return exact bytes,
MIME types, and headers; unknown/traversal paths fail closed; starting, browsing,
and refreshing extract or write no asset. Test the shell installer in temporary
prefixes on macOS and Linux and install/run the Homebrew formula on an ephemeral
macOS runner. Extracted and installed binaries must pass the same packaged smoke,
security, and lifecycle checks.

### Execution and release proof

Every pull request runs the existing formatting, test, Clippy, and release-build
checks plus semantic/inmutability conformance, HTTP security, a fast lifecycle
suite, automated browser flows and axe, dependency/license policy, and static
asset/dependency size checks. Every release candidate additionally builds all
Cargo Dist targets and runs the complete packaged security, lifecycle,
performance/resource, size, archive, installer, minimum/latest-browser, and
manual-accessibility matrices.

Produce a schema-validated `dashboard-verification.json` containing the commit,
lockfile, tool versions, targets, browser versions, artifact hashes, commands,
measurements, results, and responsible maintainer, plus a human-readable Markdown
rendering. Retain it as CI evidence and attach both representations permanently
to the GitHub Release. The publication job must depend technically on a complete,
passing, maintainer-approved manifest; a handwritten checklist or general
confidence is not release proof.
